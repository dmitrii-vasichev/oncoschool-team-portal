# Docs Policy

## Цель

Поддерживать документацию компактной, актуальной и предсказуемой по структуре.

## Область действия

Правила распространяются на все Markdown-файлы (`*.md`) в репозитории.

## Базовая структура

- Корень репозитория: только `README.md`, `CHANGELOG.md`, `CLAUDE.md`.
- `docs/README.md`: карта документации.
- `docs/plans/`: рабочие планы.
- `docs/plans/archive/`: завершённые и исторические планы.
- `docs/runbooks/`: операционные инструкции (интеграции, чеклисты, инциденты).
- `docs/ai/`: гайды и материалы для AI-ассистентов.

## Статусы документов

- `active`: документ поддерживается и используется.
- `archive`: историческая справка, не источник текущих требований.
- `deprecated`: устаревший документ, подлежит архивированию или удалению.

## Правила именования

- Имена файлов — в `UPPER_SNAKE_CASE.md` или `kebab-case.md` по смыслу раздела.
- Названия должны быть предметными, без временных имен вроде `new.md`, `temp.md`.
- Для связанных документов использовать единый префикс (например, `MEETINGS_*`).

## Жизненный цикл документов

1. Новый рабочий план создаётся в `docs/plans/`.
2. После завершения задачи план переносится в `docs/plans/archive/` в течение 7 дней.
3. Если документ не обновлялся 90+ дней и не используется, он кандидат на архив/удаление.

## Политика change-фрагментов (`.changes/`)

- Папка `.changes/` сохраняется как источник данных для сборки `CHANGELOG.md`.
- Формат и обязательные поля задаются `.changes/README.md` и `.changes/_template.md`.
- Запрещено удалять `.changes/*.md`, пока используется `make change-build`.
- Ротация: раз в месяц проверять фрагменты; перенос/чистка возможны только вместе с обновлением скриптов `scripts/change_add.py` и `scripts/change_build.py`.

## Контроль качества документации

- Раз в месяц проводить docs-audit:
  - битые ссылки;
  - дублирование содержания;
  - устаревшие инструкции.
- Любое перемещение документов должно сопровождаться обновлением ссылок в `README.md`, `CLAUDE.md`, `docs/README.md`.
